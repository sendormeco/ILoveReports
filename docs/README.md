# Документация ILoveReports

Документация описывает архитектуру проекта после доработки под тему:

> Web application to generate comprehensive reports from local data using open-source LLMs via Ollama.

## Ключевая идея проекта

Backend теперь построен вокруг классов:

- `AbstractTextGenerator`;
- `AbstractExporter`;
- `PluginRegistry`;
- `ReportService`;
- `ApplicationContainer`.

Это позволяет заменять Ollama на другой генератор и DOCX на другой экспортёр без переписывания бизнес-логики.

## Как устроена генерация отчёта

1. Пользователь заполняет форму в React.
2. Frontend отправляет данные на `POST /generate`.
3. FastAPI принимает `ReportRequest`.
4. `GenerateReportUseCase` вызывает `ReportService`.
5. `ReportService` строит промпт через `ReportPromptBuilder`.
6. `ReportService` выбирает генератор через `PluginRegistry`.
7. Генератор возвращает текст отчёта.
8. Backend возвращает `GeneratedReport`.
9. Frontend показывает отчёт пользователю.

## Как устроен экспорт

### DOCX

```http
POST /generate-docx
```

Используется совместимый endpoint для скачивания Word-файла.

### Markdown

```http
POST /export/markdown
```

Используется общий endpoint для плагинного экспорта.

## Главные классы

| Класс | Назначение |
|---|---|
| `ReportService` | Оркестрирует генерацию и экспорт. |
| `PluginRegistry` | Хранит доступные плагины и выдаёт их по имени. |
| `ApplicationContainer` | Создаёт и связывает зависимости. |
| `ReportPromptBuilder` | Строит промпт для LLM из данных пользователя. |
| `OllamaGenerator` | Запускает локальную Ollama через `subprocess`. |
| `StaticGenerator` | Позволяет проверять проект без Ollama. |
| `DocxExporter` | Создаёт `.docx` файл. |
| `MarkdownExporter` | Создаёт `.md` файл. |

## Что исправлено в диаграммах взаимодействия

В `interaction_diagrams.md` исправлены проблемные места:

- все пользовательские сценарии начинаются от пользователя;
- sequence-диаграммы заканчиваются видимым результатом для пользователя;
- вместо неподдерживаемого `hide footbox` используется `mirrorActors: false`;
- после каждой диаграммы добавлено пояснение: что происходит и за что отвечает каждый прямоугольник;
- сценарий ошибки Ollama вынесен отдельно от успешной генерации;
- добавлены объяснения для `subprocess`, `OllamaGenerator`, `PluginRegistry`, `ReportService`, export-classes и других элементов.

## Если кратко:

Можно сказать так:

> Мы доработали проект до плагинной архитектуры. Генераторы текста и экспортёры документов теперь реализуют общие интерфейсы и регистрируются в `PluginRegistry`. `ReportService` не зависит от конкретной LLM или формата файла, а выбирает нужный плагин по имени. `ApplicationContainer` собирает зависимости. Диаграммы взаимодействия показывают полный сценарий от действия пользователя до результата, а после каждой диаграммы дано пояснение роли каждого компонента.
