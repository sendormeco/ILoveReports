# Пользовательские сценарии ILoveReports

Этот файл описывает основные сценарии работы приложения ILoveReports после перехода на плагинную архитектуру. Приложение генерирует отчёты по локальным данным пользователя, может использовать разные генераторы текста и экспортировать результат в разные форматы.

## Участники системы

| Участник | Роль в системе |
|---|---|
| **Пользователь** | Заполняет данные отчёта в интерфейсе, запускает генерацию, смотрит результат, скачивает файл и открывает историю текущей сессии. |
| **Генератор текста** | Заменяемый плагин, который получает готовый промпт и возвращает текст отчёта. Основной вариант — `OllamaGenerator`, тестовые варианты — `StaticGenerator` и `MockGenerator`, дополнительный вариант — `HuggingFaceGenerator`. |
| **Экспортёр документа** | Заменяемый плагин, который превращает готовый текст отчёта в файл. Сейчас реализованы `DocxExporter` для `.docx` и `MarkdownExporter` для `.md`. |
| **Разработчик** | Может добавлять новые генераторы и экспортёры, не переписывая основную бизнес-логику приложения. |

## UC-01. Создание текста отчёта

**Цель сценария:** пользователь получает сгенерированный текст отчёта на основе введённых данных.

**Предусловия:**

- backend запущен;
- пользователь открыл форму создания отчёта;
- пользователь заполнил основные поля: цель, ход работы, результаты, вывод и данные титульного листа;
- выбранный генератор текста зарегистрирован в `PluginRegistry`.

**Основной поток:**

1. Пользователь вводит данные отчёта в `React ReportForm`.
2. Frontend вызывает функцию `generateComprehensiveReport(data)`.
3. Frontend-слой подготавливает payload для backend-а.
4. `reportApi.ts` отправляет HTTP-запрос `POST /generate`.
5. FastAPI endpoint получает объект `ReportRequest`.
6. Backend получает use case `GenerateReportUseCase` через `ApplicationContainer`.
7. `GenerateReportUseCase` вызывает `ReportService.generate_report(payload)`.
8. `ReportService` передаёт данные в `ReportPromptBuilder`.
9. `ReportPromptBuilder` собирает промпт для генератора текста.
10. `ReportService` обращается к `PluginRegistry` и выбирает нужный генератор.
11. Выбранный генератор вызывает метод `generate(prompt)`.
12. Генератор возвращает текст отчёта.
13. Backend формирует `GeneratedReport` и возвращает JSON с текстом отчёта и названием генератора.
14. Frontend показывает готовый отчёт пользователю в предпросмотре.
15. Отчёт добавляется в историю текущей сессии на frontend-е.

**Альтернативные потоки:**

- Если Ollama недоступна, backend возвращает ошибку `502`.
- Если выбран неизвестный генератор, backend возвращает ошибку `400`.
- Если для разработки выбран `static` или `mock`, отчёт создаётся без запуска локальной LLM.

**Результат:** пользователь видит сгенерированный отчёт в интерфейсе.

**Если кратко:**

> Это главный сценарий приложения. Пользователь вводит локальные данные, frontend отправляет их на backend, backend строит промпт, выбирает генератор через реестр плагинов и возвращает готовый текст отчёта.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `ReportForm` | Форма на frontend-е, где пользователь заполняет данные отчёта. |
| `generateComprehensiveReport` | Frontend use case, который преобразует данные формы в запрос для backend-а. |
| `reportApi.ts` | Сервис frontend-а, который отправляет HTTP-запросы на backend. |
| `FastAPI /generate` | Endpoint backend-а, который принимает запрос на генерацию отчёта. |
| `ReportRequest` | DTO с данными для генерации: цель, процесс, результаты, вывод, тон и генератор. |
| `ApplicationContainer` | Контейнер зависимостей, который создаёт use case, сервисы и реестр плагинов. |
| `GenerateReportUseCase` | Сценарий приложения для генерации текста отчёта. |
| `ReportService` | Главный сервис, который связывает построение промпта, выбор генератора и получение результата. |
| `ReportPromptBuilder` | Класс, который собирает промпт для LLM из данных пользователя. |
| `PluginRegistry` | Реестр, который хранит доступные генераторы и отдаёт нужный по имени. |
| `AbstractTextGenerator` | Общий интерфейс для всех генераторов текста. |
| `GeneratedReport` | Результат генерации, который возвращается frontend-у. |

## UC-02. Экспорт отчёта в DOCX

**Цель сценария:** пользователь скачивает готовый отчёт в формате `.docx`.

**Предусловия:**

- пользователь уже сгенерировал отчёт или выбрал отчёт из истории текущей сессии;
- в backend-е зарегистрирован экспортёр `DocxExporter`;
- frontend передаёт данные титульного листа и текст отчёта.

**Основной поток:**

1. Пользователь нажимает кнопку скачивания DOCX в `ReportPreview`.
2. Frontend вызывает `downloadReportAsDocx(report)`.
3. Frontend формирует `GenerateDocxPayload`.
4. `reportApi.ts` отправляет HTTP-запрос `POST /generate-docx`.
5. FastAPI endpoint получает `DocxRequest`.
6. Backend получает `GenerateDocxUseCase` через `ApplicationContainer`.
7. `GenerateDocxUseCase` вызывает экспорт через `ReportService`.
8. `ReportService` обращается к `PluginRegistry` и получает `DocxExporter`.
9. `DocxExporter` создаёт DOCX-файл и возвращает `ExportedDocument`.
10. Backend отдаёт файл через `StreamingResponse`.
11. Frontend получает `Blob`.
12. Браузер скачивает `.docx`-файл.

**Альтернативные потоки:**

- Если `DocxExporter` не зарегистрирован, backend возвращает ошибку `400`.
- Если при создании файла произошла внутренняя ошибка, backend возвращает ошибку `500`.

**Результат:** пользователь получает готовый файл отчёта в формате DOCX.

**Если кратко:**

> Экспорт DOCX реализован как отдельный плагин. `ReportService` не создаёт Word-файл напрямую, а получает экспортёр из `PluginRegistry` и вызывает общий метод `export()`.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `ReportPreview` | Экран предпросмотра отчёта, откуда пользователь запускает скачивание. |
| `downloadReportAsDocx` | Frontend use case, который готовит данные для экспорта DOCX. |
| `GenerateDocxPayload` | Объект на frontend-е с заголовком, автором, организацией, датой и текстом отчёта. |
| `FastAPI /generate-docx` | Endpoint для обратной совместимости с frontend-ом, который всегда экспортирует в DOCX. |
| `DocxRequest` | DTO backend-а с данными для создания документа. |
| `GenerateDocxUseCase` | Use case, который запускает экспорт именно в DOCX. |
| `ReportService` | Вызывает нужный экспортёр через реестр плагинов. |
| `PluginRegistry` | Возвращает экспортёр `DocxExporter` по имени `docx`. |
| `DocxExporter` | Плагин, который создаёт Word-документ. |
| `ExportedDocument` | Объект с байтами файла, именем файла, MIME-типом и расширением. |
| `StreamingResponse` | Ответ FastAPI, через который backend отдаёт файл браузеру. |
| `Blob` | Объект файла на frontend-е, который браузер скачивает как DOCX. |

## UC-03. Просмотр истории отчётов текущей сессии

**Цель сценария:** пользователь может повторно открыть отчёты, созданные во время текущей работы с приложением.

**Предусловия:**

- в текущей frontend-сессии уже был создан хотя бы один отчёт;
- приложение хранит список отчётов в состоянии React.

**Основной поток:**

1. Пользователь открывает страницу истории.
2. `HistoryPage` получает массив `GeneratedReport[]` из состояния приложения.
3. Пользователь выбирает нужную карточку отчёта.
4. `App.tsx` устанавливает выбранный отчёт как текущий.
5. Пользователь возвращается к предпросмотру выбранного отчёта.
6. Пользователь может снова просмотреть или скачать выбранный отчёт.

**Ограничение текущей реализации:**

История хранится только на frontend-е в React state. Она не сохраняется в базе данных. После перезагрузки страницы история текущей сессии может быть потеряна.

**Результат:** пользователь может открыть ранее созданный отчёт в рамках текущей сессии.

**Если кратко:**

> Сейчас история отчётов — это frontend-функция. Она помогает пользователю работать с несколькими отчётами в рамках одной сессии. Постоянное хранение можно добавить позже через авторизацию и backend-базу данных.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `App.tsx` | Хранит состояние текущего отчёта и массив истории. |
| `reportHistory` | Список отчётов, созданных в текущей сессии. |
| `GeneratedReport` | Модель с текстом отчёта, датой генерации и исходными данными. |
| `HistoryPage` | Страница, которая показывает карточки ранее созданных отчётов. |
| `handleSelectReport` | Функция, которая выбирает отчёт из истории и открывает его в предпросмотре. |
| `ReportPreview` | Экран, где пользователь снова видит выбранный отчёт. |

## UC-04. Добавление скриншотов для предпросмотра

**Цель сценария:** пользователь прикрепляет локальные изображения и видит их в интерфейсе рядом с отчётом.

**Предусловия:**

- пользователь находится на форме создания отчёта;
- пользователь выбрал файлы изображений на своём компьютере.

**Основной поток:**

1. Пользователь добавляет изображения через интерфейс формы.
2. `ReportForm` сохраняет выбранные файлы в локальное состояние React.
3. Frontend создаёт временные object URL для предпросмотра изображений.
4. `ReportPreview` показывает скриншоты во вкладке предпросмотра.
5. Пользователь визуально проверяет скриншоты вместе с отчётом.

**Ограничение текущей реализации:**

Скриншоты сейчас остаются на frontend-е. Backend их не загружает и не вставляет автоматически в DOCX-файл.

**Возможное развитие:**

- добавить backend endpoint для загрузки изображений;
- сохранять изображения в файловом хранилище;
- расширить `DocxExporter`, чтобы он вставлял скриншоты в DOCX;
- связать скриншоты с конкретным отчётом пользователя.

**Результат:** пользователь видит прикреплённые скриншоты в интерфейсе.

**Если кратко:**

> Скриншоты сейчас реализованы как UI-функция для предпросмотра. Это не ломает архитектуру, потому что позже можно добавить отдельный backend use case для загрузки файлов и расширить DOCX-экспорт.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `ReportForm` | Принимает файлы изображений от пользователя. |
| `screenshots: File[]` | Массив выбранных пользователем файлов в модели отчёта. |
| `URL.createObjectURL` | Создаёт временную ссылку на локальный файл для предпросмотра в браузере. |
| `ReportPreview` | Показывает изображения пользователю. |
| Будущий backend upload use case | Может быть добавлен позже для сохранения скриншотов на сервере. |
| Будущий DOCX image insertion | Возможное расширение `DocxExporter` для вставки изображений в документ. |

## UC-05. Просмотр доступных плагинов

**Цель сценария:** пользователь или разработчик видит, какие генераторы текста и экспортёры доступны в текущей конфигурации backend-а.

**Предусловия:**

- backend запущен;
- плагины зарегистрированы в `ApplicationContainer.plugin_registry()`.

**Основной поток:**

1. Frontend или API-клиент отправляет запрос `GET /plugins`.
2. FastAPI вызывает `ApplicationContainer.plugin_registry()`.
3. `ApplicationContainer` создаёт или возвращает `PluginRegistry`.
4. `PluginRegistry` собирает каталог зарегистрированных генераторов и экспортёров.
5. Backend возвращает список доступных плагинов.
6. Клиент может использовать имя плагина в `/generate` или `/export/{exporter_name}`.

**Результат:** пользователь или разработчик понимает, какие плагины доступны для выбора.

**Если кратко:**

> Endpoint `/plugins` показывает плагинную архитектуру снаружи. Через него можно увидеть, какие генераторы и экспортёры зарегистрированы в приложении.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `GET /plugins` | API endpoint для получения списка доступных плагинов. |
| `ApplicationContainer` | Создаёт реестр и регистрирует плагины согласно настройкам приложения. |
| `PluginRegistry` | Хранит генераторы и экспортёры. |
| `catalog()` | Метод, который возвращает список плагинов в удобном виде. |
| `PluginInfo` | Frontend-модель с названием и описанием плагина. |
| `getAvailableReportPlugins` | Frontend use case для получения списка плагинов. |

## UC-06. Экспорт отчёта в Markdown

**Цель сценария:** пользователь скачивает готовый отчёт в формате `.md`.

**Предусловия:**

- пользователь уже сгенерировал или выбрал отчёт;
- в backend-е зарегистрирован `MarkdownExporter`;
- клиент обращается к универсальному endpoint-у экспорта.

**Основной поток:**

1. Пользователь выбирает экспорт в Markdown.
2. Frontend или API-клиент отправляет `POST /export/markdown`.
3. FastAPI получает имя экспортёра из URL: `markdown`.
4. `ExportReportUseCase` вызывает `ReportService.export_report(payload, "markdown")`.
5. `ReportService` обращается к `PluginRegistry`.
6. `PluginRegistry` возвращает `MarkdownExporter`.
7. `MarkdownExporter` формирует `.md`-файл.
8. Экспортёр возвращает `ExportedDocument`.
9. Backend отдаёт файл через `StreamingResponse`.
10. Пользователь получает Markdown-файл.

**Альтернативные потоки:**

- Если экспортёр с именем `markdown` не зарегистрирован, backend возвращает ошибку `400`.
- Если во время экспорта произошла ошибка, backend возвращает ошибку `500`.

**Результат:** пользователь получает отчёт в формате Markdown.

**Если кратко:**

> Markdown-экспорт показывает расширяемость архитектуры. Сценарий почти такой же, как для DOCX, но используется другой экспортёр. Основная бизнес-логика при этом не меняется.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `POST /export/markdown` | Универсальный endpoint экспорта, где формат выбирается через URL. |
| `ExportReportUseCase` | Use case для экспорта отчёта в выбранный формат. |
| `ReportService` | Запрашивает нужный экспортёр и запускает экспорт. |
| `PluginRegistry` | Возвращает `MarkdownExporter` по имени `markdown`. |
| `MarkdownExporter` | Плагин, который создаёт Markdown-файл. |
| `ExportedDocument` | Результат экспорта с байтами файла и метаданными. |
| `StreamingResponse` | Отдаёт `.md`-файл клиенту. |

## UC-07. Заготовка авторизации пользователя

**Цель сценария:** показать место будущей авторизации в интерфейсе приложения.

**Текущее состояние:**

- страницы входа и регистрации есть на уровне frontend-интерфейса;
- полноценной backend-авторизации пока нет;
- история отчётов не привязана к аккаунту пользователя.

**Возможное развитие:**

1. Добавить backend-модель пользователя.
2. Добавить `LoginUseCase` и `RegisterUseCase`.
3. Реализовать хранение токенов на frontend-е.
4. Сохранять отчёты пользователя в базе данных.
5. Привязать историю отчётов к конкретному аккаунту.

**Результат после будущей доработки:** пользователь сможет входить в аккаунт и видеть свою постоянную историю отчётов.

**Если кратко:**

> Авторизация сейчас является заготовкой на frontend-е. Она отделена от генерации отчётов и плагинной архитектуры, поэтому её можно добавить позже без переписывания генераторов и экспортёров.

**За что отвечают основные элементы:**

| Элемент | Объяснение |
|---|---|
| `LoginPage` | Страница входа пользователя. |
| `RegisterPage` | Страница регистрации пользователя. |
| `user` в `App.tsx` | Временное состояние пользователя на frontend-е. |
| Будущий `LoginUseCase` | Use case для входа пользователя. |
| Будущий `RegisterUseCase` | Use case для регистрации пользователя. |
| Будущая backend auth layer | Слой backend-а для проверки логина, пароля и токенов. |
| Будущая база данных | Нужна для постоянного хранения пользователей и истории отчётов. |

## Краткое объяснение общей логики

Главная идея use case-ов заключается в том, что пользовательские действия проходят через frontend, затем через API и use case-и backend-а. Backend не привязан жёстко к одному генератору или одному формату файла. Вместо этого он использует `PluginRegistry`, который позволяет выбрать нужный генератор текста или экспортёр документа по имени.

Именно поэтому архитектура стала более расширяемой: чтобы добавить новый генератор или новый формат экспорта, нужно создать новый класс-плагин и зарегистрировать его в контейнере, а не переписывать все сценарии приложения.
